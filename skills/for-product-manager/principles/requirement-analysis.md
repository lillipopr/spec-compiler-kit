# 需求分析原则

## 概述
需求分析原则是产品经理进行需求分析时的指导准则，帮助正确理解和处理需求。

## 核心原则

```
问题驱动 > 用户视角 > 需求验证 > 边界清晰
```

## 1. 问题驱动原则

### 1.1 原则说明

**先理解问题，再定义解决方案**

### 1.2 5Why分析法

```
现象 → Why 1 → Why 2 → Why 3 → Why 4 → Why 5 → 根本原因
```

**示例**：
```
现象：用户说"我要一匹更快的马"

Why 1：为什么要更快的马？
→ 为了更快到达目的地

Why 2：为什么要更快到达目的地？
→ 为了节省时间

Why 3：为什么要节省时间？
→ 效率太低，影响工作

**根本原因**：提高效率，快速移动
**解决方案**：汽车，不是更快的马
```

### 1.3 问题 vs 方案

```
❌ 用户：我要一个XX功能
✅ 用户：我遇到XX问题

❌ PM：好的，我给你做个XX功能
✅ PM：你遇到什么问题？让我理解一下
```

### 1.4 问题定义框架

```markdown
## 问题定义

### 问题陈述
- 现状：XXX
- 期望：XXX
- 差距：XXX

### 问题影响
- 影响用户：XXX
- 影响业务：XXX
- 紧迫程度：XXX

### 问题原因
- 表面原因：XXX
- 深层原因：XXX
- 根本原因：XXX
```

## 2. 用户视角原则

### 2.1 原则说明

**从用户视角理解需求，不是从技术或业务视角**

### 2.2 用户语言 vs 技术语言

```
❌ 技术语言
"需要增加缓存层优化性能"

✅ 用户语言
"页面打开太慢，希望更快"
```

### 2.3 场景化理解

```markdown
## 场景化需求理解

### 5W1H分析
- Who：谁在使用？
- When：什么时候用？
- Where：在哪里用？
- What：在做什么？
- Why：为什么做？
- How：怎么做？
```

### 2.4 用户角色区分

| 用户角色 | 需求特点 | 处理方式 |
|---------|---------|----------|
| **最终用户** | 使用体验 | 优先满足 |
| **决策者** | 采购决策 | 平衡考虑 |
| **管理者** | 管理需求 | 评估必要性 |
| **技术运维** | 技术需求 | 评估可行性 |

## 3. 需求验证原则

### 3.1 原则说明

**不轻信需求描述，必须验证真实性**

### 3.2 需求验证层次

```
表面需求 → 真实需求 → 潜在需求
```

### 3.3 验证方法

| 方法 | 适用场景 | 准确度 |
|-----|---------|--------|
| **用户访谈** | 深度理解需求 | 高 |
| **问卷调研** | 广泛验证需求 | 中 |
| **数据分析** | 发现真实行为 | 高 |
| **原型测试** | 验证方案可行性 | 高 |
| **A/B测试** | 验证效果 | 最高 |

### 3.4 验证框架

```markdown
## 需求验证

### 验证问题
1. 这个需求真的存在吗？
2. 有多少人遇到这个问题？
3. 问题有多严重？
4. 用户会怎么解决？
5. 用户愿意为此付出什么？

### 验证方法
- 用户访谈：XX人
- 问卷调研：XX人
- 数据分析：XX
- 原型测试：XX人

### 验证结论
- 需求真实性：高/中/低
- 需求紧迫性：高/中/低
- 建议行动：XXX
```

## 4. 边界清晰原则

### 4.1 原则说明

**明确需求的边界，防止范围蔓延**

### 4.2 In/Out Scope

```markdown
## 需求边界

### In Scope（本期包含）
- 功能1：XXX
- 功能2：XXX

### Out of Scope（本期不含）
- 功能X：XXX（原因：需求不明确）
- 功能Y：XXX（原因：技术难度大）
- 功能Z：XXX（原因：优先级低）

### 未来考虑
- 功能A：XXX（V2.0考虑）
- 功能B：XXX（需求验证后考虑）
```

### 4.3 防止范围蔓延

| 信号 | 说明 | 应对 |
|-----|------|------|
| **"顺便加个..."** | 典型范围蔓延 | 记录到需求池 |
| **"这个很简单"** | 往往不简单 | 评估工作量 |
| **"用户可能需要"** | 没有验证 | 先验证再做 |

### 4.4 边界管理流程

```
确定边界 → 记录边界 → 沟通边界 → 守住边界 → 变更管理
```

## 5. 优先级原则

### 5.1 原则说明

**不是所有需求都同等重要，必须排序**

### 5.2 优先级判断

```
用户价值 × 紧迫程度 × 实现成本 = 优先级
```

### 5.3 MoSCoW分类

| 分类 | 说明 | 开发顺序 |
|-----|------|----------|
| **Must have** | 必须有 | 第1批 |
| **Should have** | 应该有 | 第2批 |
| **Could have** | 可以有 | 第3批 |
| **Won't have** | 不会有 | 不做 |

### 5.4 优先级动态调整

```markdown
## 优先级调整触发条件

### 提升优先级
- 用户强烈要求
- 数据显示重要
- 竞品压力
- 战略调整

### 降低优先级
- 验证后不重要
- 资源不足
- 技术风险高
```

## 6. 可测试性原则

### 6.1 原则说明

**需求必须可以转化为可测试的验收标准**

### 6.2 可测试性判断

```markdown
## 可测试性检查

### 不可测试的需求
❌ "优化用户体验"
❌ "提升性能"
❌ "设计美观"

### 可测试的需求
✅ "注册流程从3步减少到2步"
✅ "页面加载时间<2秒"
✅ "用户满意度>4.5分"
```

### 6.3 验收标准编写

```gherkin
Given <前置条件>
When <触发动作>
Then <预期结果>
```

## 7. 可追溯性原则

### 7.1 原则说明

**需求从来源到上线全程可追溯**

### 7.2 需求追溯链

```
需求来源 → 需求记录 → 用户故事 → 开发任务 → 测试用例 → 上线功能
```

### 7.3 追溯信息

| 追溯项 | 记录内容 |
|-------|----------|
| **需求来源** | 谁、何时、为什么 |
| **需求变更** | 变更历史、原因 |
| **需求关联** | 关联需求、依赖 |
| **实现状态** | 当前状态、负责人 |

## 8. 需求完整性原则

### 8.1 原则说明

**需求必须完整，不遗漏关键信息**

### 8.2 需求完整性检查清单

```markdown
## 需求完整性检查

### 背景信息
- [ ] 业务背景
- [ ] 用户痛点
- [ ] 业务价值

### 需求定义
- [ ] 目标用户
- [ ] 使用场景
- [ ] 功能描述

### 验收标准
- [ ] 正常场景
- [ ] 边界场景
- [ ] 异常场景

### 约束条件
- [ ] 技术约束
- [ ] 资源约束
- [ ] 时间约束
```

## 9. 需求可行性原则

### 9.1 原则说明

**需求必须在技术、资源、时间上可行**

### 9.2 可行性评估

| 维度 | 评估问题 | 结论 |
|-----|---------|------|
| **技术可行性** | 技术能实现吗？ | 可行/不可行 |
| **资源可行性** | 资源足够吗？ | 可行/不可行 |
| **时间可行性** | 时间够吗？ | 可行/不可行 |
| **经济可行性** | 投入产出合理吗？ | 可行/不可行 |

### 9.3 不可行需求处理

```markdown
## 不可行需求处理

### 明确不可行原因
- 技术限制：XXX
- 资源限制：XXX
- 时间限制：XXX

### 提出替代方案
- 方案A：XXX（优：XXX，劣：XXX）
- 方案B：XXX（优：XXX，劣：XXX）

### 与需求方沟通
- 解释原因
- 提供方案
- 达成共识
```

## 10. 沟通共识原则

### 10.1 原则说明

**需求必须与相关方达成共识**

### 10.2 相关方识别

| 相关方 | 关注点 | 沟通重点 |
|-------|--------|----------|
| **业务方** | 业务价值 | 需求价值、优先级 |
| **用户** | 使用体验 | 用户体验、反馈 |
| **开发** | 实现难度 | 技术方案、工作量 |
| **测试** | 质量保证 | 验收标准、测试点 |
| **运维** | 稳定可靠 | 稳定性、监控 |

### 10.3 共识达成流程

```
需求收集 → 需求分析 → 方案设计 → 方案评审 → 达成共识
```

## 原则应用示例

```markdown
## 需求分析示例

### 用户需求
"我要一个实时通讯功能"

### 应用原则分析

#### 1. 问题驱动
- 用户遇到什么问题？
- 为什么需要实时通讯？
- 现在的方案有什么问题？

#### 2. 用户视角
- 什么场景下需要？
- 对实时性要求多高？
- 需要哪些功能？

#### 3. 需求验证
- 真的有这个需求吗？
- 有多少用户需要？
- 验证方式：用户访谈、数据分析

#### 4. 边界清晰
- In Scope：基础聊天功能
- Out Scope：语音、视频、群聊（后续版本）

#### 5. 优先级
- 用户价值：高
- 实现成本：中
- 优先级：P1

#### 6. 可测试性
- 消息延迟 < 1秒
- 消息到达率 > 99%

### 最终需求
经过分析和验证，定义出清晰的需求文档
```

## 常见误区

- ❌ 用户说要就要
- ❌ 不验证就做
- ❌ 边界不清晰
- ❌ 所有需求优先级相同
- ❅ 需求不断变化，没有版本概念
