# 产品规划工作流

## 概述
产品规划是将产品愿景和目标转化为具体执行计划的过程，通过功能拆分、优先级排序、路线图制定，指导团队有序执行。

## 适用场景

- 产品战略规划
- 版本迭代规划
- 季度/年度规划
- 资源分配决策

## 工作流程

```
目标设定 → 功能收集 → 功能拆分 → 优先级排序 → 资源评估 → 路线图制定
```

## 1. 目标设定

### 1.1 目标层级

```
愿景 → 战略目标 → 季度目标 → 版本目标
```

### 1.2 OKR方法

```markdown
## OKR设定

### Objective（目标）
- 定性描述：XXX

### Key Results（关键结果）
- KR1：XXX（可量化）
- KR2：XXX（可量化）
- KR3：XXX（可量化）
```

**示例**：

```markdown
## Q1 OKR

### Objective
提升用户活跃度和留存率

### Key Results
- KR1：DAU从10万提升到15万（+50%）
- KR2：次日留存从40%提升到50%（+25%）
- KR3：7日留存从20%提升到30%（+50%）
```

### 1.3 目标分解

```
年度目标 → 季度目标 → 月度目标 → 版本目标
```

| 层级 | 时间跨度 | 示例 |
|-----|---------|------|
| 年度目标 | 1年 | DAU达到100万 |
| 季度目标 | 3个月 | DAU从10万到50万 |
| 月度目标 | 1个月 | DAU从10万到15万 |
| 版本目标 | 2-4周 | 上线XX功能 |

## 2. 功能收集

### 2.1 功能来源

| 来源 | 处理方式 |
|-----|----------|
| **用户反馈** | 归纳分析，识别共性 |
| **数据洞察** | 数据分析，发现机会 |
| **竞品动态** | 竞品分析，差异化 |
| **业务需求** | 业务目标，价值评估 |
| **团队想法** | 创新探索，可行性 |
| **战略规划** | 战略方向，拆解执行 |

### 2.2 功能池管理

建立统一的功能池：

```markdown
## 功能池

### 待评估
- 功能A（来源：用户反馈）
- 功能B（来源：数据洞察）

### 待优先级排序
- 功能C（已完成初步评估）
- 功能D（已完成初步评估）

### 已规划
- 功能E（已纳入V1.2）
- 功能F（已纳入V1.3）

### 已完成
- 功能G（V1.1已上线）

### 已拒绝
- 功能H（不符合产品定位）
```

### 2.3 功能记录格式

```markdown
## 功能记录

### 基本信息
- 功能ID：FEA-001
- 功能名称：XXX
- 来源：XXX
- 提出时间：XXX

### 功能描述
- 功能概述：XXX
- 用户价值：XXX
- 业务价值：XXX

### 初始评估
- 战略匹配：高/中/低
- 用户价值：高/中/低
- 实现难度：难/中/易

### 状态
- 当前状态：待评估/待排序/已规划/已完成/已拒绝
```

## 3. 功能拆分

参见：[功能拆分方法论](../methodology/feature-breakdown.md)

### 3.1 拆分原则

```
大功能 → 独立可交付 → 用户价值完整 → 符合INVEST
```

### 3.2 拆分方法

| 方法 | 适用场景 |
|-----|----------|
| 按用户流程 | 用户流程清晰的功能 |
| 按功能层次 | 有核心-扩展结构的功能 |
| 按数据对象 | 管理类功能 |
| 按复杂度（MVP） | 技术复杂的功能 |

### 3.3 拆分示例

```
大功能：会员系统

拆分：
1. 会员开通（浏览 → 选择 → 支付）
2. 会员权益（查看权益 → 使用权益）
3. 会员管理（续费 → 自动续费 → 取消）
4. 会员成长（等级 → 升级 → 特权）
```

## 4. 优先级排序

参见：[优先级排序方法论](../methodology/priority-ranking.md)

### 4.1 评估模型

使用RICE模型：

```
RICE = (Reach × Impact × Confidence) / Effort
```

### 4.2 评估维度

| 维度 | 权重 | 说明 |
|-----|------|------|
| 用户价值 | 25% | 对用户的价值 |
| 商业价值 | 25% | 对业务的贡献 |
| 战略价值 | 15% | 对长期战略的意义 |
| 实现成本 | 20% | 开发和维护成本 |
| 风险 | 10% | 技术和市场风险 |
| 紧急度 | 5% | 时间敏感度 |

### 4.3 排序流程

```
功能列表 → 维度评分 → 计算分数 → 排序 → 团队讨论 → 确认
```

### 4.4 排序结果

```markdown
## 优先级排序结果

### P0（必须做）
1. 功能A（RICE分数：XXX）
   - 理由：XXX

2. 功能B（RICE分数：XXX）
   - 理由：XXX

### P1（应该做）
1. 功能C（RICE分数：XXX）
   - 理由：XXX

### P2（可以做）
1. 功能D（RICE分数：XXX）
   - 理由：XXX
```

## 5. 资源评估

### 5.1 资源盘点

```markdown
## 资源盘点

### 人力资源
- 产品：X人
- 设计：X人
- 前端：X人
- 后端：X人
- 测试：X人

### 时间资源
- 季度：13周
- 可用迭代：X个（每迭代2周）

### 预算资源
- 营销预算：XXX
- 基础设施：XXX
```

### 5.2 工作量估算

| 功能 | 产品 | 设计 | 前端 | 后端 | 测试 | 总计 |
|-----|------|------|------|------|------|------|
| 功能A | 2天 | 3天 | 5天 | 8天 | 3天 | 21天 |
| 功能B | 1天 | 2天 | 3天 | 5天 | 2天 | 13天 |
| 功能C | 3天 | 5天 | 8天 | 10天 | 4天 | 30天 |

### 5.3 资源平衡

```
总工作量 ÷ 可用资源 × 时间 = 需要的迭代数
```

**示例**：

```
总工作量：100人天
可用资源：5人
每迭代时间：2周（10个工作日）
每迭代容量：5人 × 10天 = 50人天
需要迭代：100 ÷ 50 = 2个迭代
```

### 5.4 资源调整策略

| 情况 | 调整策略 |
|-----|----------|
| 工作量 > 资源 | 降低P2功能优先级，减少范围 |
| 工作量 < 资源 | 增加P2功能，或提升质量 |
| 某角色瓶颈 | 调整功能顺序，或增加资源 |

## 6. 路线图制定

### 6.1 路线图类型

| 类型 | 时间跨度 | 粒度 |
|-----|---------|------|
| **战略路线图** | 1-3年 | 季度 |
| **版本路线图** | 3-6个月 | 月/版本 |
| **迭代路线图** | 1-2个月 | 周/迭代 |

### 6.2 路线图制定方法

```
目标 → 功能 → 优先级 → 资源 → 时间 → 路线图
```

### 6.3 版本规划

```markdown
## 版本规划

### V1.0（MVP - 3月）
**目标**：验证核心需求

**功能**：
- P0：用户注册登录
- P0：核心功能A
- P0：核心功能B

### V1.1（6月）
**目标**：完善体验

**功能**：
- P1：优化功能A体验
- P1：新增功能C
- P2：新增功能D

### V1.2（9月）
**目标**：扩展功能

**功能**：
- P1：新增功能E
- P2：新增功能F
- P2：优化性能
```

### 6.4 迭代规划

```markdown
## 迭代规划

### Iteration 1（第1-2周）
**目标**：完成用户注册登录

**功能**：
- US-001：用户注册
- US-002：用户登录
- US-003：密码找回

**里程碑**：用户可以注册并登录

### Iteration 2（第3-4周）
**目标**：完成核心功能A

**功能**：
- US-004：功能A-基础
- US-005：功能A-进阶
- US-006：功能A-通知

**里程碑**：用户可以使用功能A
```

### 6.5 路线图可视化

使用甘特图或时间线展示：

```

─────────────────────────────────────
V1.0      V1.1          V1.2
███          ████            █████
```

## 7. 跟踪与调整

### 7.1 跟踪指标

```markdown
## 跟踪指标

### 进度跟踪
- 计划功能数：X
- 完成功能数：Y
- 完成率：Y/X%

### 质量跟踪
- Bug数：X
- 严重Bug数：Y
- 测试通过率：Z%

### 目标跟踪
- OKR完成度：X%
- 里程碑达成：X/X
```

### 7.2 调整触发

| 触发条件 | 调整动作 |
|---------|----------|
| 进度严重落后 | 削减范围，或增加资源 |
| 目标无法达成 | 调整目标，或调整计划 |
| 市场重大变化 | 重新规划，或调整优先级 |
| 用户反馈强烈 | 提升相关功能优先级 |

### 7.3 版本回顾

每个版本结束后进行回顾：

```markdown
## 版本回顾

### 目标达成
- [ ] 目标1：XXX
- [ ] 目标2：XXX

### 数据表现
- 指标1：XXX（目标：XXX）
- 指标2：XXX（目标：XXX）

### 经验总结
- 做得好的：XXX
- 需要改进的：XXX

### 下一步
- 进入V1.X
```

## 9. 常见误区

- ❌ 目标不清晰，只是为了做功能而做
- ❌ 只看价值，不看成本，资源估算不准
- ❌ 优先级只靠产品经理一人决定
- ❌ 路线图制定后就不再调整
- ❅ 规划太细，失去了灵活性
