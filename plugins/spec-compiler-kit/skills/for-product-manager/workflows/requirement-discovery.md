# 需求探索工作流

## 概述
需求探索是从模糊想法到清晰需求的转化过程，通过系统的调研和分析，确保产品决策基于真实的用户需求和市场机会。

## 适用场景

- 产品早期探索
- 新功能机会评估
- 战略方向研究
- 问题解决探索

## 工作流程

```
想法输入 → 快速判断 → 调研计划 → 信息收集 → 分析总结 → 需求定义
```

## 1. 想法输入

### 1.1 输入来源

| 来源 | 类型 | 处理方式 |
|-----|------|----------|
| **业务方** | 明确需求 | 记录并理解业务目标 |
| **用户反馈** | 零散问题 | 分类归纳，识别共性 |
| **数据洞察** | 数据异常 | 深入分析原因 |
| **竞品动态** | 市场变化 | 评估影响 |
| **团队想法** | 创新灵感 | 初步判断可行性 |

### 1.2 快速记录

使用统一格式记录想法：

```markdown
## 想法记录
- 来源：XXX
- 时间：XXX
- 核心想法：XXX
- 背景：XXX
```

## 2. 快速判断

### 2.1 判断维度（5分钟）

```
战略匹配 → 用户价值 → 实现难度 → 资源评估
```

### 2.2 快速决策矩阵

| 维度 | 高（值得深入） | 中（可以观望） | 低（暂不处理） |
|-----|-------------|-------------|-------------|
| **战略匹配** | 核心方向 | 相关方向 | 无关方向 |
| **用户价值** | 强痛点 | 中等价值 | 弱需求 |
| **实现难度** | 可行 | 需要研究 | 技术壁垒 |
| **资源要求** | 可满足 | 需要协调 | 资源不足 |

### 2.3 决策结果

- **高**：进入详细调研
- **中**：记录到需求池，等待时机
- **低**：归档，说明原因

## 3. 调研计划

### 3.1 计划要素

```markdown
## 调研计划

### 调研目标
- 核心问题：XXX
- 输出物：XXX

### 调研范围
- 市场调研
- 用户调研
- 竞品调研

### 调研方法
- 桌面研究
- 用户访谈
- 问卷调研

### 时间安排
- 第1-2天：市场、竞品调研
- 第3-4天：用户访谈
- 第5天：分析总结
```

### 3.2 并行调研

```
市场调研 ───┐
            ├──→ 综合分析
用户调研 ───┘
```

## 4. 信息收集

### 4.1 市场调研（1-2天）

参见：[市场分析方法论](../methodology/market-analysis.md)

**核心输出**：
- 市场规模评估
- 市场趋势判断
- 用户画像定义

### 4.2 竞品调研（1-2天）

参见：[竞品分析方法论](../methodology/competitor-analysis.md)

**核心输出**：
- 竞品列表
- 竞品功能对比
- 差异化机会

### 4.3 用户调研（2-3天）

**调研方法**：

| 方法 | 适用场景 | 样本量 |
|-----|---------|--------|
| **用户访谈** | 深度了解 | 5-8人 |
| **问卷调研** | 广泛验证 | 100+人 |
| **场景观察** | 了解真实行为 | 3-5人 |
| **数据分析** | 发现模式 | - |

**调研问题框架**：

```markdown
## 用户访谈提纲

### 背景
- 使用场景
- 使用频率
- 现有方案

### 痛点
- 遇到的问题
- 期望的改进
- 优先级排序

### 验证
- 方案偏好
- 支付意愿
- 使用意愿
```

### 4.4 信息收集技巧

- **多角度验证**：不同来源的信息交叉验证
- **关注异常**：异常数据可能隐藏机会
- **记录原始信息**：不要过早归纳
- **保持怀疑**：不轻信单一来源

## 5. 分析总结

### 5.1 综合分析框架

```
市场机会 + 用户需求 + 竞品格局 = 产品机会
```

### 5.2 分析维度

| 维度 | 分析问题 | 输出 |
|-----|---------|------|
| **市场吸引力** | 市场大吗？增长快吗？ | 市场评分 |
| **用户需求强度** | 痛点强吗？刚需吗？ | 需求评分 |
| **竞争格局** | 竞争激烈吗？有差异化机会吗？ | 竞争评分 |
| **团队能力** | 我们能做吗？有优势吗？ | 能力评分 |

### 5.3 机会评估矩阵

```
高吸引力
  │  ┌────────┐
  │  │ ★ 优先│
  │  └────────┘
  │
  │  ┌────────┐
  │  │ ? 观察│
  │  └────────┘
  │
  └──────────────→ 低难度
```

### 5.4 结论输出

```markdown
## 调研结论

### 核心发现
1. ...
2. ...
3. ...

### 机会评估
- 市场吸引力：高/中/低
- 用户需求强度：高/中/低
- 竞争格局：激烈/一般/蓝海
- 团队能力匹配：高/中/低

### 建议
- [ ] 进入产品设计
- [ ] 继续观察
- [ ] 暂不处理

### 风险提示
- ...
```

## 6. 需求定义

### 6.1 定义原则

```
用户语言 → 产品语言 → 验收标准
```

### 6.2 定义模板

```markdown
## 需求定义

### 需求背景
- 业务背景：XXX
- 用户痛点：XXX
- 业务价值：XXX

### 需求目标
- 核心目标：XXX
- 量化指标：XXX

### 目标用户
- 用户画像：XXX
- 使用场景：XXX

### 核心功能
- 功能1：XXX
- 功能2：XXX

### 成功指标
- 指标1：XXX
- 指标2：XXX
```

### 6.3 下一步

进入 [功能定义工作流](./feature-definition.md)，开始编写PRD。

## 7. 常见误区

- ❌ 想法来了直接做，不做调研
- ❌ 只看数据，不看用户
- ❌ 只看竞品，不看用户
- ❌ 调研太久，错失时机
- ❅ 调研了但不决策

## 8. 关键产出

| 阶段 | 产出物 |
|-----|--------|
| 想法输入 | 想法记录 |
| 调研计划 | 调研计划文档 |
| 信息收集 | 市场报告、竞品报告、用户调研报告 |
| 分析总结 | 调研结论、机会评估 |
| 需求定义 | 需求定义文档 |
