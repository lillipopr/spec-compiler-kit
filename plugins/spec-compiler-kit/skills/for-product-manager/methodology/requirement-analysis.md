# 需求分析方法论

## 概述
需求分析是从各种渠道收集需求，经过分析、整理、验证，最终转化为清晰、可执行的产品需求的过程。

## 分析框架

```
需求收集 → 需求分类 → 需求分析 → 需求验证 → 需求定义
```

## 1. 需求收集

### 1.1 收集渠道

| 渠道 | 方法 | 频率 |
|-----|------|------|
| **用户反馈** | 客服记录、用户访谈、问卷调研 | 持续 |
| **数据分析** | 行为分析、漏斗分析、留存分析 | 定期 |
| **竞品分析** | 竞品功能、用户评价 | 定期 |
| **业务需求** | 业务方沟通、战略会议 | 定期 |
| **团队洞察** | 头脑风暴、技术探索 | 不定期 |
| **行业趋势** | 行业报告、技术趋势 | 定期 |

### 1.2 收集方法

#### 用户访谈

```markdown
## 用户访谈提纲

### 开场
- 自我介绍
- 说明目的
- 承诺保密

### 基本信息
- 使用频率
- 使用场景
- 使用习惯

### 核心问题
- 遇到的问题
- 期望的功能
- 痛点排序

### 验证问题
- 方案展示
- 偏好选择
- 支付意愿

### 收尾
- 其他建议
- 感谢参与
```

**访谈技巧**：
- 多听少说
- 追问"为什么"
- 观察行为
- 记录原话

#### 问卷调研

```markdown
## 问卷设计原则

### 问题类型
1. 单选：明确选项，互斥
2. 多选：多重选择
3. 开放：自由表达

### 问题顺序
1. 简单问题在前
2. 复杂问题在后
3. 敏感问题最后

### 问卷长度
- 控制在5-10分钟
- 不超过20题
```

| **路径分析** | 发现行为模式 | 同上 |

## 2. 需求分类

### 2.1 分类维度

#### 按来源分类

| 来源 | 处理优先级 |
|-----|----------|
| 用户强需求 | 高 |
| 数据验证 | 高 |
| 业务战略 | 中高 |
| 团队想法 | 中 |
| 随意想法 | 低 |

#### 按类型分类

| 类型 | 说明 |
|-----|------|
| **功能需求** | 系统应该做什么 |
| **性能需求** | 速度、容量等 |
| **体验需求** | 易用性、美观 |
| **数据需求** | 报表、分析 |
| **合规需求** | 法律、政策 |

#### 按Kano模型分类

| 类型 | 优先级 | 说明 |
|-----|--------|------|
| **基本需求** | 高 | 必须有，没有就不满意 |
| **期望需求** | 中 | 越多越满意 |
| **兴奋需求** | 可选 | 有就很惊喜 |

## 3. 需求分析

### 3.1 5W1H分析法

| 维度 | 问题 | 目的 |
|-----|------|------|
| **What** | 是什么需求？ | 明确需求内容 |
| **Why** | 为什么需要？ | 挖掘真实意图 |
| **Who** | 谁需要？ | 明确目标用户 |
| **When** | 什么时候需要？ | 明确时间要求 |
| **Where** | 在什么场景？ | 明确使用场景 |
| **How** | 怎么实现？ | 初步评估可行性 |

### 3.2 根因分析（5Why）

```
现象：用户说"我要一匹更快的马"

Why 1：为什么要更快的马？
→ 为了更快到达目的地

Why 2：为什么要更快到达目的地？
→ 为了节省时间

Why 3：为什么要节省时间？
→ 效率太低，影响工作

**真实需求**：提高效率，快速移动
**解决方案**：汽车，不是更快的马
```

### 3.3 用户价值分析

```markdown
## 用户价值分析框架

### 用户价值
- 解决什么痛点？
- 带来什么便利？
- 节约什么成本？

### 使用频率
- 每天用？每周用？偶尔用？

### 用户覆盖
- 所有用户？部分用户？

### 紧迫程度
- 不用不行？可以等待？
```

### 3.4 业务价值分析

```markdown
## 业务价值分析框架

### 收益分析
- 增加收入？
- 降低成本？
- 提升效率？

### 成本分析
- 开发成本
- 维护成本
- 机会成本

### ROI分析
- ROI = (收益 - 成本) / 成本
- 优先做ROI高的需求
```

### 3.5 可行性分析

```markdown
## 可行性分析

### 技术可行性
- 技术成熟度
- 团队能力
- 技术风险

### 业务可行性
- 符合战略
- 有资源
- 有商业模式

### 经济可行性
- 投入产出比
- 回收周期
```

## 4. 需求验证

### 4.1 验证维度

```
真实性 → 价值性 → 可行性 → 完整性
```

### 4.2 验证方法

#### 用户验证

| 方法 | 适用场景 | 准确度 |
|-----|---------|--------|
| **用户访谈** | 深度验证 | 高 |
| **问卷调研** | 广泛验证 | 中 |
| **原型测试** | 可用性验证 | 高 |
| **A/B测试** | 效果验证 | 高 |

#### 数据验证

```markdown
## 数据验证指标

### 现有数据
- 这个功能有没有数据支撑？
- 有多少人会遇到这个问题？

### 数据规模
- 影响多少用户？
- 占比多少？

### 数据频率
- 每天发生多少次？
```

#### 竞品验证

```markdown
## 竞品验证

### 竞品是否做？
- 如果做了：验证市场需求
- 如果没做：为什么没做？

### 竞品怎么做？
- 功能设计
- 用户体验

### 差异化机会
- 我们能做得更好吗？
- 有差异化机会吗？
```

### 4.3 MVP验证

```markdown
## MVP验证流程

### 1. 设计MVP
- 最小可行方案
- 核心价值验证

### 2. 快速实现
- 简化实现
- 不追求完美

### 3. 小范围测试
- 灰度发布
- 收集反馈

### 4. 数据分析
- 使用数据
- 用户反馈

### 5. 决策
- 继续投入？
- 调整方向？
- 放弃？
```

## 7. 常见误区

### 7.1 收集误区

- ❌ 只听用户说的，不看用户做的
- ❌ 只看数据，不看原因
- ❌ 只看数量，不看质量
- ❅ 收集了不整理

### 7.2 分析误区

- ❌ 把用户方案当需求
- ❌ 只看表面，不挖根因
- ❌ 只看价值，不看成本
- ❅ 分析了不验证

### 7.3 验证误区

- ❌ 只问用户"想要吗"
- ❌ 只做内部评审，不问用户
- ❅ 验证了不决策

### 7.4 正确做法

- ✅ 观察用户行为，不只听用户说
- ✅ 挖掘真实需求，不只看表面
- ✅ 用数据验证，不只凭感觉
- ✅ 快速验证，不拖太久
